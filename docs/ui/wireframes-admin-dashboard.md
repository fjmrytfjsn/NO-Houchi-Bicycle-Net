# Admin Dashboard — 画面構成図・低忠実度ワイヤーフレーム

## 目的

区役所職員が放置自転車の通報状況を確認し、未解除案件の回収依頼と回収結果記録までを行うための管理画面構成と低忠実度ワイヤーフレームを整理する。

今回の成果物は高忠実度UIや実装コンポーネントではなく、Issue 20 の画面設計レビューに使う画面間の関係、各画面の情報配置、操作責務を示す。

## 対象画面

1. 通報一覧
   - 役割: 全通報を状態、日時、位置で確認する起点画面
   - 表示要素: 写真サムネイル、通報日時、位置、識別情報、ステータス
   - 操作: 通報詳細を開く、状態で絞り込む

2. 未解除案件
   - 役割: `reported` または `temporary` の案件を回収依頼候補として確認する画面
   - 表示要素: 経過時間、現在ステータス、位置、識別情報、写真サムネイル
   - 操作: 回収依頼画面へ進む

3. 回収依頼
   - 役割: 対象案件に依頼メモを付け、回収依頼状態へ更新する画面
   - 表示要素: 対象通報の概要、依頼メモ、依頼登録確認
   - 操作: 回収依頼を登録し、ステータスを `collection_requested` にする

4. 回収結果記録
   - 役割: 回収業者からの現地結果を受けて、案件の最終結果を記録する画面
   - 表示要素: 対象通報の概要、回収依頼状態、結果メモ、結果選択
   - 操作: `collected` または `not_found_on_collection` を記録する

## 補助導線

- 通報詳細
  - 役割: 通報一覧から個別通報の写真、位置、識別情報、操作履歴を確認する補助導線
  - 表示要素: 写真、通報日時、位置、識別情報、現在ステータス、履歴
  - 操作: 未解除案件として確認する、一覧へ戻る

## 画面遷移

```mermaid
flowchart LR
  A["通報一覧<br>全通報を状態・日時で確認"] --> B["通報詳細<br>写真・位置・識別情報・履歴を確認"]
  A --> C["未解除案件<br>reported / temporary を回収候補として確認"]
  C --> D["回収依頼<br>対象案件に依頼メモを付けて登録"]
  D --> E["回収結果記録<br>回収完了 / 現地で現物なしを記録"]
  E --> F["完了状態<br>collected / not_found_on_collection"]
  B --> C
```

## 画面別ワイヤーフレーム

### 通報一覧

```text
┌──────────────────────────────────────────────┐
│ 管理者ダッシュボード                         │
├──────────────────────────────────────────────┤
│ フィルター: 状態 [すべて] 日時 [期間] 位置   │
├──────────────────────────────────────────────┤
│ サマリー: 全件 / 未解除 / 回収依頼中         │
├──────────────────────────────────────────────┤
│ 写真 | 通報日時 | 位置 | 識別情報 | 状態     │
│ 写真 | 通報日時 | 位置 | 識別情報 | 状態     │
│ 写真 | 通報日時 | 位置 | 識別情報 | 状態     │
├──────────────────────────────────────────────┤
│ 操作: 詳細を見る / 未解除案件へ              │
└──────────────────────────────────────────────┘
```

- 通報一覧は職員の起点画面とし、全件確認と状態絞り込みを優先する。
- 一覧行には写真、通報日時、位置、識別情報、ステータスを並べる。
- `reported` / `temporary` の案件は未解除案件へ進める導線を表示する。

### 未解除案件

```text
┌──────────────────────────────────────────────┐
│ 未解除案件                                   │
├──────────────────────────────────────────────┤
│ 絞り込み: reported / temporary               │
├──────────────────────────────────────────────┤
│ 経過時間 | 位置 | 現在ステータス             │
├──────────────────────────────────────────────┤
│ 写真 | 識別情報 | 通報日時 | 経過時間        │
│ 写真 | 識別情報 | 通報日時 | 経過時間        │
│ 写真 | 識別情報 | 通報日時 | 経過時間        │
├──────────────────────────────────────────────┤
│ 操作: 回収依頼へ進む                         │
└──────────────────────────────────────────────┘
```

- 未解除案件は回収依頼候補の確認に集中する。
- 通報からの経過時間を見える位置に置き、回収判断を補助する。
- 対象案件を選ぶと回収依頼画面に遷移する。

### 回収依頼

```text
┌──────────────────────────────────────────────┐
│ 回収依頼                                     │
├──────────────────────────────────────────────┤
│ 対象概要: 写真 / 位置 / 識別情報 / 状態      │
├──────────────────────────────────────────────┤
│ 依頼メモ                                     │
│ [                                         ]  │
├──────────────────────────────────────────────┤
│ 確認: collection_requested に更新            │
├──────────────────────────────────────────────┤
│ [キャンセル]                  [回収依頼登録] │
└──────────────────────────────────────────────┘
```

- 回収依頼は、対象通報の取り違えを避けるため概要を先頭に表示する。
- 依頼メモは任意入力とし、登録後は `collection_requested` に更新する。
- 登録操作とキャンセル操作を明確に分ける。

### 回収結果記録

```text
┌──────────────────────────────────────────────┐
│ 回収結果記録                                 │
├──────────────────────────────────────────────┤
│ 対象概要: 写真 / 位置 / 回収依頼状態         │
├──────────────────────────────────────────────┤
│ 結果選択                                     │
│ ( ) 回収完了                                 │
│ ( ) 現地で現物なし                           │
├──────────────────────────────────────────────┤
│ 結果メモ                                     │
│ [                                         ]  │
├──────────────────────────────────────────────┤
│ [キャンセル]                    [結果を記録] │
└──────────────────────────────────────────────┘
```

- 回収結果は `collected` または `not_found_on_collection` のどちらかを選ぶ。
- 結果メモは、回収業者からの現地連絡内容を残すために配置する。
- 記録後は案件を完了状態として扱う。

## 参照API

| 画面 | API | 用途 |
| --- | --- | --- |
| 通報一覧 | `GET /api/reports` | 管理者向けの通報一覧を取得する |
| 通報詳細 | `GET /api/reports/{id}` | 個別通報の詳細を取得する |
| 回収依頼 | `POST /api/reports/{id}/collection-request` | 未解除案件を回収依頼対象にする |
| 回収結果記録 | `PATCH /api/reports/{id}/collection-result` | 回収完了または現地で現物なしを記録する |

## UI/UX メモ

- 状態ラベルは色だけに依存せず、`reported`、`temporary`、`collection_requested` などのテキストを併記する。
- 通報一覧と未解除案件では、職員が優先度を判断できるように通報日時と経過時間を近くに表示する。
- 回収依頼と回収結果記録は、誤操作を避けるため対象通報の概要を確認できる構成にする。
- ブラックリスト、ヒートマップ、保管・返還管理、詳細分析は今回の試作品スコープ外とし、将来構想として扱う。
