# Admin Dashboard UI 評価基準と改善目標

## 目的

管理者ダッシュボードの UI 状態を、感想ベースではなく再採点可能な数値で評価する。
この文書は `apps/admin-dashboard` の現状を基準に、評価基準、現状点、目標点、75 点到達までの改善 Backlog を 1 セットで定義する。

対象は現行試作品スコープの管理画面に限定し、ヒートマップ、高度分析、ブラックリスト管理、回収後の保管・返還管理は評価対象外とする。

## 採点ルール

- 各指標は `1.0` から `5.0` まで `0.5` 刻みで採点する。
- 評価基準は 1-5 段階のルーブリックで固定し、移行途中の状態は中間値 `0.5` を使って採点する。
- 100 点換算は `指標点 x 20` とする。
- 総合点は各指標の 100 点換算値の平均を四捨五入して算出する。
- 再採点時は、各指標に 1 行の採点理由を必ず残す。

### ルーブリック

| 段階 | 基準 |
| --- | --- |
| 1 | 実務で明確な支障がある。主要操作が迷いやすく、判断材料が不足している。 |
| 2 | 動作はするが非効率。主要操作や優先判断に余計な確認が必要。 |
| 3 | 最低限成立。慣れた利用者なら運用可能だが、判断支援や導線が弱い。 |
| 4 | 実務で十分使いやすい。主要操作と優先判断が短い手数で成立する。 |
| 5 | 明確に洗練されている。重要情報、操作優先度、狭幅対応まで高い完成度で整っている。 |

## 現状スコアカード

### 総合点

- 現状: `57 / 100`
- 次の目標: `75 / 100`
- 判定基準: 各指標を再採点し、総合点が 75 点以上、かつ Must 項目がすべて完了していること。

## 指標別評価

### 1. 情報設計

- 目的: 要対応案件、経過時間、回収対象候補を先頭で把握できるようにする。
- 現状点: `3.0 / 5.0` = `60 / 100`
- 現状の根拠:
  - `通報一覧` は状態フィルターと基本サマリーを持つが、優先判断に必要な `24時間超過` や `回収対象件数` が上部に出ていない。
  - `回収依頼候補` でも件数サマリーがなく、一覧を読まないと全体状況がつかめない。
- 目標点: `4.0 / 5.0` = `80 / 100`
- 判定条件:
  - 上部に `要対応件数` `24時間超過件数` `回収対象件数` を表示する。
  - 各画面が「確認画面」ではなく「判断画面」として機能する。
- 主な改善項目:
  - Must-01 優先度サマリー追加
  - Must-02 回収依頼候補画面の件数整理

### 2. 発見性

- 目的: 画面ごとの主要操作をテキスト探索なしで見つけられるようにする。
- 現状点: `2.5 / 5.0` = `50 / 100`
- 現状の根拠:
  - 一覧画面と詳細画面は主要操作がテキストリンク中心で、主操作が埋もれやすい。
  - 詳細画面では、ステータスに応じた次の推奨操作が弱く、画面を見ただけで次の行動が分かりにくい。
- 目標点: `4.0 / 5.0` = `80 / 100`
- 判定条件:
  - 主要操作がボタンまたは強調リンクとして一貫表示される。
  - 詳細画面と一覧画面で、ステータスごとの次アクションが即判別できる。
- 主な改善項目:
  - Must-03 主操作のボタン化と優先表示
  - Should-02 詳細画面の推奨次アクション表示

### 3. 一覧操作性

- 目的: 一覧から対象案件を探す、並べる、絞る操作を短時間で完了できるようにする。
- 現状点: `2.0 / 5.0` = `40 / 100`
- 現状の根拠:
  - `通報一覧` は状態フィルターのみで、期間、候補状態、キーワード、並び替えがない。
  - `回収依頼候補` も `reported全件` と `回収対象のみ` の切り替えしかなく、優先処理のための整列手段が不足している。
- 目標点: `4.0 / 5.0` = `80 / 100`
- 判定条件:
  - `期間` `候補状態` `キーワード` で絞り込める。
  - `通報日時順` と `経過時間順` の並び替えができる。
  - 優先案件を一覧上で早く見つけられる。
- 主な改善項目:
  - Must-04 一覧フィルター拡張
  - Should-01 並び替え追加

### 4. ステータス可読性

- 目的: 状態の意味、緊急度、次工程を短時間で判別できるようにする。
- 現状点: `3.5 / 5.0` = `70 / 100`
- 現状の根拠:
  - 状態バッジ自体はあり、色だけに依存していない。
  - 一方で、`reported` と `collection_requested` などの業務上の意味差分や、候補状態の違いは視覚的に十分整理されていない。
- 目標点: `4.0 / 5.0` = `80 / 100`
- 判定条件:
  - 状態バッジに加え、候補状態や処理段階の補助ラベルが一貫して表示される。
  - 自動候補、手動候補、手動除外が一覧上で区別できる。
- 主な改善項目:
  - Must-05 回収依頼候補画面での候補状態の視覚差分

### 5. フォーム安全性

- 目的: 回収依頼と結果記録の入力を、対象確認と変更結果の見通しを持って実行できるようにする。
- 現状点: `3.0 / 5.0` = `60 / 100`
- 現状の根拠:
  - 対象概要と入力欄は分かれており、最低限の確認はできる。
  - ただし、変更後ステータスの見通し、成功時の完了感、詳細画面への戻り方が弱く、安心感が足りない。
- 目標点: `4.0 / 5.0` = `80 / 100`
- 判定条件:
  - 実行前に変更後ステータスと対象 ID を確認できる。
  - 実行後に成功状態と次アクションが分かる。
  - 対象外ステータスの時は、処理不能理由が明確に示される。
- 主な改善項目:
  - Must-06 フォーム画面での変更後ステータス明示
  - Should-03 成功後フィードバック強化

### 6. レスポンシブ・視認性

- 目的: ノート PC の狭い横幅でも、主要情報と操作の視認性を維持する。
- 現状点: `2.5 / 5.0` = `50 / 100`
- 現状の根拠:
  - 1 カラム切り替えはあるが、テーブル自体の狭幅時制御が弱い。
  - サマリー、一覧、詳細の主従差もまだ弱く、管理画面としての視線誘導が不足している。
- 目標点: `3.5 / 5.0` = `70 / 100`
- 判定条件:
  - 狭幅時に横スクロールまたはカード化で情報欠落を防げる。
  - 主要 KPI と主要操作が視線の上流に残る。
- 主な改善項目:
  - Must-07 狭幅時の一覧可読性改善
  - Could-01 視覚トーンの再整理
  - Could-02 余白と情報密度の微調整

## KPI

| KPI | 現状 | 目標 | 計測方法 |
| --- | --- | --- | --- |
| 一覧から回収依頼開始までのクリック数 | `3-4` | `2-3` | 通報一覧から回収依頼画面到達までのクリック数を数える |
| 回収対象判定に必要な確認項目数 | `4以上` | `2-3` | 職員が候補判定に見る UI 要素数を数える |
| 主要操作の初見判別率 | 未計測 | `90%以上` | 5 人程度の簡易レビューで主要操作を迷わず特定できる割合を記録する |

## 再採点ルール

1. Must 項目が 1 つでも未完了なら、総合点 75 点以上でも「目標未達」と扱う。
2. 再採点時は各指標の `点数` と `採点理由 1 行` を必ず残す。
3. 75 点未達の場合は、未達の原因を `未完了の Must` または `判定条件の不足` のどちらかで記録する。
4. API 変更が必要になった改善項目は、この文書の Backlog からは外さず、実装タスク側で別タスク化する。
