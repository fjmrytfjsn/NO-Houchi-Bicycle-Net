# Codex 依頼テンプレート

この文書は、Codex に動作確認やマニュアル作成を短く依頼するための再利用テンプレートをまとめたものです。

## 使い方

- `対象`、`URL`、`確認範囲`、`保存先` だけ差し替えて使う。
- スクリーンショット付きでまとめたい場合は、テンプレート内の `SS` 指示を残す。
- 動作確認とマニュアル作成を同時にやる場合は、まず「まとめて依頼」を使う。

## まとめて依頼

```text
[対象] の動作確認とマニュアル作成を並行してやって。
動作確認は [確認範囲] を対象にして、SS付きで結果をまとめてほしい。
マニュアルは [想定読者] 向けで、画面ごとに手順と注意点を整理し、SS付きでわかりやすくしてほしい。
保存先は [保存先] を使って。
必要なら起動、検証、差分確認、commit、push、PR作成まで進めて。
```

### 記入例

```text
管理者ダッシュボードの動作確認とマニュアル作成を並行してやって。
動作確認は ログイン、通報一覧、通報詳細、未解除案件、回収依頼、回収結果記録 を対象にして、SS付きで結果をまとめてほしい。
マニュアルは 区役所職員向け で、画面ごとに手順と注意点を整理し、SS付きでわかりやすくしてほしい。
保存先は docs/testing/screenshots/2026-05-28/ を使って。
必要なら起動、検証、差分確認、commit、push、PR作成まで進めて。
```

## 動作確認だけ依頼

```text
[対象] の動作確認をして。
確認範囲は [確認範囲]。
実行環境、起動コマンド、確認手順、期待結果、実結果、既知の注意点、未実施事項をまとめて、SSも付けてほしい。
スクリーンショット保存先は [保存先]。
関連テスト、lint、typecheck、build を実行して、未実施があれば明記して。
```

### 記入例

```text
管理者ダッシュボードの動作確認をして。
確認範囲は ログイン、通報一覧、通報詳細、未解除案件、回収依頼、回収結果記録。
実行環境、起動コマンド、確認手順、期待結果、実結果、既知の注意点、未実施事項をまとめて、SSも付けてほしい。
スクリーンショット保存先は docs/testing/screenshots/2026-05-28/。
関連テスト、lint、typecheck、build を実行して、未実施があれば明記して。
```

## マニュアル作成だけ依頼

```text
[対象] の運用マニュアルを作って。
想定読者は [想定読者]。
[対象画面] を含めて、各画面ごとに 目的、開き方、操作手順、確認ポイント、注意点 を整理してほしい。
SSを付けて、保存先は [保存先] にして。
既存ドキュメントの索引更新が必要ならあわせて対応して。
```

### 記入例

```text
管理者ダッシュボードの運用マニュアルを作って。
想定読者は 区役所職員または運用担当。
ログイン、通報一覧、通報詳細、未解除案件、回収依頼、回収結果記録 を含めて、各画面ごとに 目的、開き方、操作手順、確認ポイント、注意点 を整理してほしい。
SSを付けて、保存先は docs/testing/screenshots/2026-05-28/ にして。
既存ドキュメントの索引更新が必要ならあわせて対応して。
```

## 指示を足したいときの短句

- 実 API 前提で確認して
- seed データを使って確認して
- エラー系は主対象外で、重大なものだけ補足して
- 既知不具合は修正せず記録だけして
- PR は日本語タイトル・本文で作って
- スクリーンショット名は `admin-01-...` のように連番で揃えて

## 保存先の決め方

- スクリーンショット: `docs/testing/screenshots/YYYY-MM-DD/`
- 動作確認レポート: `docs/testing/manual-verification-YYYY-MM-DD.md`
- 運用マニュアル: `docs/operations/[対象]-manual.md`

## この repo での依頼時に添えると安定する情報

- 対象アプリ: `apps/admin-dashboard` / `apps/owner-web` / `backend`
- 起動 URL: 例 `http://localhost:3002`
- 確認したい主要フロー
- seed 利用可否
- スクリーンショット保存先の日付
- マニュアルの想定読者
