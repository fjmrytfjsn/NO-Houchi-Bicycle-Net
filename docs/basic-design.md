# 基本設計仕様書

**プロジェクト:** NO-Houchi Bicycle Net
**バージョン:** 1.3

## 1. 概要・目的

本書は、NO-Houchi Bicycle Net の価値を伝えるための最低限の試作品に向けた基本設計を記述する。

本試作品では、放置自転車への対応を以下の一連の流れとして示す。

- サポーターが自転車を発見し、Android アプリから通報する
- 持ち主が QR から警告内容を確認し、解除を行う
- 持ち主が戻らない場合は、管理者が回収業者への回収依頼を行う
- 回収結果を「回収完了」または「現地で現物なし」として記録する

この試作品の目的は、通報から解決または回収対応までを一つの運用としてつなげ、ソリューションの有効性を短時間で理解できるようにすることである。

## 2. システム構成

本システムは以下の 4 コンポーネントで構成される。

- **サポーター用 Android アプリ**: 放置自転車の発見、撮影、位置情報付き通報、QR/ポイント要素の提示を行う
- **持ち主用 Web アプリ（QRアクセス）**: マーカーの QR からアクセスし、警告確認、仮解除、本解除を行う
- **管理者用ダッシュボード**: 通報一覧の確認、未解除案件の把握、回収依頼、回収結果の記録を行う
- **バックエンド API サーバー**: 通報登録、通報参照、解除状態管理、回収依頼状態管理、操作履歴記録を行う

## 3. 試作品で説明する主要ストーリー

本試作品では、以下の 3 つのストーリーを主要シナリオとして扱う。

1. **通報 → 持ち主が解除して解決**
2. **通報 → 持ち主が戻らない → 回収依頼 → 正常回収**
3. **通報 → 持ち主が戻らない → 回収依頼 → 現地で自転車なし（未回収記録）**

## 4. 機能要件

### 4.1 試作品の必須機能

#### サポーター用 Android アプリ

- 放置自転車を撮影して通報できる
- 通報時に位置情報と識別情報を送信できる
- 現地での利用イメージとして QR 要素を扱える
- ポイント機能の存在を簡潔に示せる

#### 持ち主用 Web アプリ

- QR から対象の警告内容を確認できる
- 持ち主が仮解除を申請できる
- 一定時間経過後に本解除できる

#### 管理者用ダッシュボード

- 通報一覧を確認できる
- 未解除案件を確認できる
- 一定時間内に持ち主による解除が行われなかった通報を、回収依頼対象として扱える
- 回収結果を「回収完了」または「現地で現物なし」として記録できる

#### バックエンド API サーバー

- Android アプリからの通報登録を受け付ける
- 管理画面向けに通報一覧・詳細を提供する
- Owner Web 向けに警告参照、仮解除、本解除を提供する
- 回収依頼状態および回収結果を管理する
- 最小限の操作履歴を保持する

### 4.2 将来構想として扱う機能

以下は構想上は重要だが、今回の試作品では必須要件に含めない。

- ブラックリスト管理の本実装
- ヒートマップや高度分析
- 回収後の保管・返還管理
- ポイントやクーポンの詳細インセンティブ設計
- 不正検知の高度化
- 本番運用レベルの監視・可用性
- マイクロサービス前提の拡張構成

## 5. 非機能要件

- **応答性**: モバイル端末で利用した際に、主要画面や主要操作で大きな待ち時間が発生しないこと
- **通信保護**: 全通信を HTTPS で行う
- **操作履歴**: 通報、仮解除、本解除、回収依頼、回収結果更新の最低限の記録を残す
- **プライバシー配慮**: サポーターの行動履歴を過剰に保持せず、通報地点など必要最小限の情報のみ扱う

## 6. アーキテクチャ / 技術スタック

- **フロントエンド**
  - Android アプリ: Android (Kotlin)
  - Web アプリ: React / Next.js
- **バックエンド**: Node.js
- **データベース**: PostgreSQL
- **ストレージ**: 証拠写真保存用ストレージ
- **外部連携**: OCR 連携は利用候補とするが、試作品では詳細要件を固定しない

## 7. 主要インターフェース

基本設計上の主要インターフェースは以下の 4 系統とする。

- **通報登録**: Android アプリから写真・位置情報・識別情報を送信する
- **通報参照**: 管理者用ダッシュボードが通報一覧・詳細を取得する
- **解除フロー**: 持ち主用 Web アプリが `GET /api/owner/markers/{code}`、`POST /unlock-temp`、`POST /unlock-final` を利用する
- **回収運用**: 管理画面が回収依頼登録と回収結果更新を行う

## 8. データモデル（主要エンティティ）

- **Marker**: QR で参照される識別子
- **BicycleReport**: 通報情報、位置情報、画像、状態を保持する
- **MoveDeclaration**: 仮解除・本解除に関する情報を保持する
- **User**: サポーターまたは管理者を表す
- **CollectionRequest**: 回収依頼および回収結果の履歴を保持する

## 9. 状態遷移

試作品で明示する主要状態は以下とする。

- `reported`
- `temporary`
- `resolved`
- `collection_requested`
- `collected`
- `not_found_on_collection`

想定する最小の状態遷移は以下の 3 系統である。

- `reported → temporary → resolved`
- `reported → collection_requested → collected`
- `reported → collection_requested → not_found_on_collection`

## 10. データフロー（運用シナリオ）

### シナリオ A: 持ち主が解除して解決する場合

1. サポーターが放置自転車を撮影し、位置情報と識別情報を付けて通報する
2. サーバーが通報を登録し、状態を `reported` にする
3. 持ち主が QR を読み、警告内容を確認する
4. 持ち主が仮解除を行い、状態が `temporary` になる
5. 一定時間経過後に持ち主が本解除を行い、状態が `resolved` になる

### シナリオ B: 回収依頼後に正常回収される場合

1. サポーターが通報し、状態が `reported` になる
2. 一定時間内に持ち主による解除が行われない
3. 管理者が未解除案件として確認し、回収依頼を登録する
4. 状態が `collection_requested` になる
5. 回収業者が現地対応を行う
6. 回収業者の現地結果を受けて、管理者が管理画面で「回収完了」を記録する
7. 状態が `collected` になり、案件は完了扱いとなる

### シナリオ C: 回収依頼後に現地で自転車が見つからない場合

1. サポーターが通報し、状態が `reported` になる
2. 一定時間内に持ち主による解除が行われない
3. 管理者が未解除案件として確認し、回収依頼を登録する
4. 状態が `collection_requested` になる
5. 回収業者が現地対応を行う
6. 現地で自転車が見つからなかった場合、管理者が管理画面で「現地で現物なし」を記録する
7. 状態が `not_found_on_collection` となり、未回収記録として残る

## 11. 将来拡張の扱い

`not_found_on_collection` は試作品では運用記録のための状態として扱う。
将来的には、不正対策やブラックリスト連携の候補として活用する可能性があるが、今回の試作品ではその連携を必須要件に含めない。

## 12. 運用・監視

- 最低限の操作ログを保持する
- システム異常時に原因追跡できるログを残す
- 本番運用向けの詳細な監視設計は将来フェーズで拡張する
