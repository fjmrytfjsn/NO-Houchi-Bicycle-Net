# 基本設計仕様書

**プロジェクト:** NO-Houchi Bicycle Net
**バージョン:** 1.2

## 1. 概要・目的

大阪府北区における放置自転車問題を解決するためのシビックテック・プラットフォームの基本設計を記述する。
市民（サポーター）のスマートフォンとAI-OCR技術を活用し、放置自転車の早期発見・警告を行う。また、QRコードを用いた「タイムラグ解除機能」により、持ち主の確実な移動を促し、不正なタグ廃棄（逃げ得）をブラックリスト機能で防止する。

### 1.1 価値説明のための最小アウトカム

- **誰のどの課題を解決するか**: 北区の放置自転車に対し、サポーター＋AI-OCR＋QR再スキャンで「逃げ得なく」早期是正し、誤撤去を抑止する。
- **Before/Afterと効果指標**: 現状フローと導入後フローを比較し、滞留時間短縮・誤撤去抑止・担当者工数削減を示す。測定軸は「検知成功率/誤検知率」「検知→通知レイテンシ」「対応開始までの時間」「巡回・回収工数削減率」。
- **コア機能の最小セット**: AI-OCR通報、タイムラグ付き警告、QR再スキャンによる本解除とクーポン、ブラックリスト自動化、管理マップ＋回収指示。
- **信頼性・安心材料**: 非破壊マーカー、位置履歴非保持＋HTTPS＋署名付きURL、匿名化ヒートマップ、稼働目標99.9%。
- **実証スコープの限定**: 拠点1、デバイス/カメラ少数、通知チャネルはSlack/Webhookのいずれか1系統、外部API連携は最小限。
- **早期に示す成果物**: 動くデモ（1現場想定）、設定ファイル例（エリア・閾値・通知先）、検知ログとKPIサマリ（検知率・誤検知率・レイテンシ）。

## 2. システム構成

本システムは以下の4コンポーネントで構成される。

- **サポーター用 Android アプリ**: 放置自転車の発見、OCRスキャン、ポイント管理を行う。
- **持ち主用 Web アプリ（QRアクセス）**: マーカーのQRからアクセスし、警告確認および解除申請（移動宣言）を行う。
- **管理者用ダッシュボード**: 北区役所・回収業者が使用。マップ表示、ブラックリスト管理、回収指示を行う。
- **バックエンド API サーバー**: データ処理、画像保存、外部OCR連携、ステータス管理を行う。

加えて、次フェーズで以下のサブレイヤーを導入する。

- **外部連携レイヤー**:
  - 警察連携API（盗難届との照合を想定した連携インターフェース）
  - 店舗連携API（クーポン発行・利用の地域連携）
- **サーバーサイド解析レイヤー**:
  - 画像品質判定（ピンぼけ、暗所、QR欠損）
  - 常習犯検知（防犯登録番号・画像特徴量ベースの照合）
  - 不正検知/監査ログ分析（短時間連続通報など）
- **データ分析レイヤー**:
  - 通報の時空間分析、ホットスポット可視化、行政向けエビデンス出力
- **オフライン同期マネージャー**:
  - 圏外時の通報キューイング、再送、重複排除

## 3. 機能要件（高レベル）

### サポーター機能

- **AI-OCR通報**: カメラで車体番号・防犯登録番号等を読み取り、位置情報と共に登録。
- **インセンティブ**: 情報量（OCR読取数）に応じたポイント付与、ランキング表示。
- **ジオフェンシング通報制限**: 駐輪場・私有地など通報禁止エリアでは通報不可とする。
- **撮影品質アシスト**: ガイドフレーム表示とリアルタイム品質判定により、再撮影を促す。
- **タイムラグ通報ロジック**: 短時間駐輪の誤通報抑制のため、一定時間経過後に確定通報する。
- **安全装着ガイド**: マーカー取り付け時に安全位置（ブレーキ干渉回避など）を画面で誘導する。

### 持ち主機能

- **警告確認**: QRコード読み取りで警告ステータスを表示。放置自転車画像とOCRテキストも確認可能。
- **タイムラグ解除（仮解除）**: Web上で移動申請を行う。即時解除はせず、一定時間（例:15分）経過後の移動確認をもって本解除とする。
- **本解除のQR再スキャン**: 仮解除後、同一マーカーのQRコードをカメラで再スキャンすることで本解除を実行。不正な解除を防止。
- **クーポンの獲得**: 本解除完了時に商店街クーポンを自動発行。
- **安心情報表示**: マーカーの非破壊性（巻き付け式）と行政委託業務である旨を明示し、トラブルを抑制する。

### 管理者機能

- **放置マップ**: ヒートマップおよびピン表示による現状把握。
- **ブラックリスト管理**: 解除申請なく消失した自転車（ID特定済み）をリスト化。次回発見時に即時撤去アラートを出す。
- **回収指示**: 撤去対象リストの出力およびステータス更新。
- **常習犯自動アラート**: 同一防犯登録番号の違反回数が閾値超過時、即時撤去候補として通知する。
- **盗難車フロー分岐**: 盗難照合一致時は業者回収フローではなく警察連携フローへ分岐する。

## 4. 非機能要件

- **可用性**: 99.9% 目標（24時間稼働、深夜回収対応のため）。
- **レスポンス**: QR読み込みからWeb表示まで < 2秒（モバイル回線想定）。
- **セキュリティ**:
  - 全通信のHTTPS化。
  - サポーターの位置情報（行動履歴）は保持せず、通報地点のみ保存。
  - 画像データの非公開化（署名付きURLでのみ管理者へ表示）。
- **拡張性**: 将来的な他自治体展開を考慮したマイクロサービス構成（推奨）。
- **経済的持続性**:
  - KPIとして「巡回コスト削減率」「回収1件あたり工数削減」「削減費の運用費充当率」を管理する。
- **法的安全性/低リスク性**:
  - マーカーは非破壊（粘着剤不使用）かつ緊急時に破断可能な仕様とする。
- **匿名化**:
  - 行政向け統計には個人特定情報を含めず、ヒートマップ生成時に匿名化処理を適用する。
- **物理的耐久性**:
  - マーカー素材は防水性とUV耐性を満たすこと。

## 5. アーキテクチャ / 技術スタック

- **フロントエンド**:
  - App: Android (Kotlin)
  - Web: React / Next.js (SPA)
- **バックエンド**: Node.js (NestJS / Express)
- **データベース**:
  - RDB: PostgreSQL (ユーザー、マーカー、トランザクション管理)
  - NoSQL: Firestore (オプション: リアルタイム状態同期用)
- **ストレージ**: AWS S3 互換ストレージ (証拠写真保存)
- **外部サービス**: Azure Microsoft Foundry OCR API (文字認識)

## 6. データモデル（主要エンティティ）

- **Bicycle (放置自転車)**: ID, Status(警告中/仮解除/撤去対象), OCR_Data(車体番号等), Image_URL, Location
- **Supporter (サポーター)**: ID, Point, Rank
- **Blacklist (要注意リスト)**: ID, Feature_Vector, Past_Offenses
- **GeofenceZone (通報制限エリア)**: ZoneID, PolygonCoordinates, ZoneType(駐輪場/私有地/禁止区画)
- **AuditLog (監査ログ)**: LogID, UserID, DeviceID, IPAddress, ActionType, RiskScore
- **CleanupRecord (回収実績)**: RecordID, ReportID, ContractorID, DisposalLocation, CostSavedEstimate

## 7. データフロー（運用シナリオ）

1.  **発見**: サポーターが自転車を撮影し、OCRで番号を読み取って送信。
2.  **警告**: サーバーがデータを登録し、ステータスを「警告中」にする。
3.  **分岐**:
    - **A (正規ルート)**: 持ち主がQRを読み、「移動申請」ボタンを押す。システムは「仮解除」モードへ移行。一定時間後に移動が確認されれば解決。
    - **B (不正ルート)**: 持ち主がタグを捨てて逃げる。後日、ブラックリスト照合により「逃走車」として登録される。
    - **C (放置継続)**: 持ち主が現れず、一定時間経過後に管理者が「回収指示」を出す。
4.  **解決**: 回収または移動完了により、ポイントが確定・付与される。

## 8. 運用・監視

- **ログ集約**: CloudWatch Logs / ELK Stack
- **アラート**:
  - システム異常（DB接続エラー等）
  - 業務アラート（ブラックリスト該当車両の検知時）
  - 不正通報アラート（短時間連続、同一端末の異常行動）
